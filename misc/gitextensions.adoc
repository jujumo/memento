== Goal

Run the Windows portable build of *GitExtensions* on Linux through Wine, with a dedicated `WINEPREFIX` so it doesn't pollute the default one. SSH key passphrases are loaded from the Linux terminal because Wine GUI prompts ignore numpad digits.

Layout used in this guide:

[cols="<1m,<2", width=80%]
|===
| Path | Purpose

| ~/Applications/wine-gitext/ | dedicated `WINEPREFIX`
| ~/Downloads/gitext/         | GitExtensions portable zip
| ~/.local/bin/gitext         | shell wrapper launching the app
| ~/.local/bin/gitext-ssh-add | helper to load an SSH key per session
| ~/.cache/wine-gitext/       | runtime state (agent env, log)
|===

Versions verified working at time of writing:

* Git for Windows *2.54.0*
* GitExtensions Portable *5.1.1.17970*
* Wine 9.x+

== Prerequisites

. Host packages:
+
[source,bash]
----
sudo apt install wine winetricks winbind unzip bsdmainutils
----
+
NOTE: `bsdmainutils` provides `script(1)`, used by `gitext-ssh-add` to give mingw `ssh-add` a pty.

. Linux SSH keys present at `~/.ssh/id_rsa` (and any other you intend to use).

== Wine prefix

A `WINEPREFIX` is the directory Wine uses as a self-contained "virtual Windows installation": fake `C:\` (`drive_c/`), the Windows registry (`system.reg`, `user.reg`), drive-letter map (`dosdevices/`), and per-user `AppData`. If `$WINEPREFIX` is unset, Wine uses `~/.wine`.

Using a *dedicated* prefix for GitExtensions instead of the default:

* *isolation* — a bad installer or winetricks verb can't break other Wine apps;
* *reproducibility* — `rm -rf ~/Applications/wine-gitext` and rebuild from this doc, no collateral;
* *per-app tuning* — different apps may need different .NET versions, fonts, or Windows version emulation;
* *easy backup* — just tar the prefix.

Every `wine` invocation selects its prefix via the env var (the `gitext` wrapper sets it automatically, so all child processes — `GitExtensions.exe`, `git.exe`, `ssh.exe`, `ssh-agent.exe` — end up inside the same isolated tree).

Create an isolated prefix dedicated to GitExtensions:

[source,bash]
----
export WINEPREFIX="$HOME/Applications/wine-gitext"
mkdir -p "$WINEPREFIX"
wineboot --init
----

Install the runtimes GitExtensions needs (.NET 8 Desktop and core fonts):

[source,bash]
----
WINEPREFIX="$HOME/Applications/wine-gitext" winetricks -q \
  corefonts dotnetdesktop8 dotnetdesktop9
----

== Git for Windows

Download the *standalone* installer (not the portable one) from https://gitforwindows.org/[gitforwindows.org] — releases live on https://github.com/git-for-windows/git/releases[GitHub]:

[source,bash]
----
mkdir -p ~/Downloads
wget -P ~/Downloads \
  https://github.com/git-for-windows/git/releases/download/v2.54.0.windows.1/Git-2.54.0-64-bit.exe

WINEPREFIX="$HOME/Applications/wine-gitext" wine ~/Downloads/Git-2.54.0-64-bit.exe
----

In the installer, select:

* *SSH Option*: `OpenSSH` (use the bundled mingw `ssh.exe`, **not** PuTTY/Plink)
* *Path Option*: `Cmd`
* *Editor*: VIM (or your choice)
* Leave the other defaults as-is.

== GitExtensions portable

Download the portable `.zip` from https://github.com/gitextensions/gitextensions/releases[GitExtensions releases] and extract to `C:\GitExtensions` inside the prefix:

[source,bash]
----
mkdir -p ~/Downloads/gitext
wget -P ~/Downloads/gitext \
  https://github.com/gitextensions/gitextensions/releases/download/v5.1.1/GitExtensions-Portable-x64-5.1.1.17970-939c0edba.zip

mkdir -p "$WINEPREFIX/drive_c/GitExtensions"
unzip ~/Downloads/gitext/GitExtensions-Portable-x64-*.zip \
      -d "$WINEPREFIX/drive_c/GitExtensions"
----

NOTE: The filename includes a build hash that changes per release. Pick the matching asset from the releases page.

Symlink your real `~/.ssh` so GitExtensions sees the same keys and `known_hosts` as Linux:

[source,bash]
----
ln -s "$HOME/.ssh" "$WINEPREFIX/drive_c/users/$USER/.ssh"
----

Optional baseline `gitconfig` inside the prefix (sets Meld as merge/diff tool):

..gitconfig (inside the wine prefix)
[source,ini]
----
[user]
	name = name
	email = name@org.com
[merge]
	tool = meld
	guitool = meld
[mergetool "meld"]
	path = C:/Program Files/Meld/Meld.exe
	cmd = "C:/Program Files/Meld/Meld.exe" "$LOCAL" "$BASE" "$REMOTE" --output "$MERGED"
[diff]
	tool = meld
	guitool = Meld
[difftool "Meld"]
	path = C:/Program Files/Meld/Meld.exe
	cmd = "C:/Program Files/Meld/Meld.exe" "$LOCAL" "$REMOTE"
[mergetool]
	keepBackup = false
----

Drop it at `~/Applications/wine-gitext/drive_c/users/$USER/.gitconfig`.

== Launcher script

The `gitext` wrapper sets `WINEPREFIX`, translates Linux paths to Wine paths, and forwards the SSH agent env (see next section) into Wine.

The script lives next to this doc: link:gitext[gitext]. Install it (run from this directory):

[source,bash]
----
install -m 0755 gitext ~/.local/bin/gitext
----

Usage:

[source,bash]
----
gitext              # opens current dir
gitext path/to/repo # opens a specific repo
----

== SSH key handling

=== The numpad problem

GitExtensions, running under Wine, shows a built-in passphrase prompt (Pageant-style) whose Wine widget *silently drops digits typed from the numeric keypad*. Any passphrase containing digits cannot be entered there.

Solution: load the key into a Wine-side mingw `ssh-agent` *before* launching GitExtensions, typing the passphrase in your *Linux* terminal (the host kernel handles keypresses normally). GitExtensions then picks up the key silently via `SSH_AUTH_SOCK`.

=== Disable autostart Pageant

Edit `~/Applications/wine-gitext/drive_c/GitExtensions/GitExtensions.settings` and set:

[source,xml]
----
<key>
  <string>autostartpageant</string>
</key>
<value>
  <string>false</string>
</value>
----

This makes GitExtensions defer to `git` + OpenSSH instead of its built-in Pageant flow.

=== The `gitext-ssh-add` helper

Why this is non-trivial — four traps stack on top of each other:

. *Linux `ssh-agent` socket ≠ mingw `ssh.exe` socket.* The Linux gnome-keyring agent speaks `AF_UNIX`; Git-for-Windows' bundled `ssh.exe` speaks Cygwin/MSYS sockets (TCP-backed magic files). So we must run a *mingw* `ssh-agent.exe` under Wine, and load keys into _that_ agent.
. *mingw `ssh-add` prompts only on a pty.* A plain stdin pipe is ignored. Wrapping the call with `script -qfc` gives it one — keystrokes then come from your Linux terminal (where the numpad works).
. *wineserver caches its environment at first launch.* If some unrelated `wine` call has already started `wineserver` with a stale `SSH_AUTH_SOCK` (e.g. inherited from gnome-keyring), every later Wine process inherits that stale value and our agent is unreachable. Mitigation: kill `wineserver` (`wineserver -k`) before starting fresh, and *export `SSH_AUTH_SOCK` before* the very first `wine` call.
. *GitExtensions sanitizes the env block when it spawns `git.exe`.* `.NET ProcessStartInfo.EnvironmentVariables` builds a fresh env block from scratch, dropping vars merely inherited from the parent — so shell-side `export SSH_AUTH_SOCK=...` is not enough. Mitigation: pin `SSH_AUTH_SOCK` in `HKCU\Environment` via `wine reg add`. Windows injects registry env into every `CreateProcess` call regardless of what GitExtensions strips.

The helper below addresses all four. It also uses a *deterministic* socket path (`~/.ssh/agent/wine.sock`, exposed inside the prefix via the `drive_c/users/$USER/.ssh` symlink) so the registry-pinned `SSH_AUTH_SOCK` value never goes stale across restarts.

The helper lives next to this doc: link:gitext-ssh-add[gitext-ssh-add]. Install it (run from this directory):

[source,bash]
----
install -m 0755 gitext-ssh-add ~/.local/bin/gitext-ssh-add
----

=== Workflow per session

From a *terminal*, just launch `gitext`. The wrapper auto-invokes `gitext-ssh-add` when stdin/stdout are ttys; if no key is loaded yet it prompts here (numpad works), if one already is it's a sub-second no-op.

[source,bash]
----
gitext path/to/repo                  # first run: prompts in this terminal
                                     # next runs: silent, agent reused
----

From a *desktop launcher / file manager* (no controlling tty), the wrapper instead probes the agent and shows a `notify-send` warning if no key is loaded — it never blocks on a prompt that has nowhere to render. In that case, open a terminal and run `gitext-ssh-add` once:

[source,bash]
----
gitext-ssh-add ~/.ssh/id_rsa         # explicit, for GUI-launch users
----

The Wine ssh-agent lives as long as its parent `wineserver`, so it stays up across multiple `gitext` invocations and dies when the prefix's wineserver exits (idle timeout, logout, or `wineserver -k`). `gitext-ssh-add` itself is idempotent: if a live agent already holds a key, re-running is a no-op.

== Known issues / FAQ

=== `Cygwin WARNING: Couldn't compute FAST_CWD pointer`

Harmless under Wine. Cygwin/MSYS tries to locate a Windows kernel pointer to speed up `getcwd()`/path conversion. That pointer doesn't exist in Wine (it's not real Windows), so Cygwin falls back to a slower compat path. Ignore.

=== Numerous `fixme:` wine messages

Normal noise from Wine stubs. None of those observed so far block GitExtensions, `git`, or `ssh`. The wrapper redirects them to `~/.cache/gitext-wine.log`.

=== `The agent has no identities` after reboot

Expected — the Wine-side `ssh-agent.exe` dies with the user session. Run `gitext-ssh-add` again.

=== `Permission denied (publickey)` from the GUI even though `gitext-ssh-add` says the key is loaded

Almost always caused by a stale `HKCU\Software\GitExtensions\gitssh` registry value. GitExtensions reads that key at startup and, if non-empty, exports it as `GIT_SSH=<value>` for every `git` child process. Git's ssh resolution order is `GIT_SSH_COMMAND` > `core.sshCommand` > `GIT_SSH` > PATH — so a leftover `Z:\usr\bin\ssh` (from the abandoned host-`ssh` experiment described in the next FAQ entry) silently overrides the working mingw `ssh.exe`, routing every push through a Linux ELF that Wine can't spawn cleanly. Symptom: `ssh_askpass: exec(/usr/bin/ssh-askpass): No such file or directory`, then `Permission denied (publickey)`.

Fix — clear it, or pin it to the mingw `ssh.exe` so future migrations don't bring back the `Z:\...` value:

[source,bash]
----
WINEPREFIX="$HOME/Applications/wine-gitext" wine reg add \
  'HKCU\Software\GitExtensions' /v gitssh /t REG_SZ \
  /d 'C:\Program Files\Git\usr\bin\ssh.exe' /f
----

Diagnosis trick when this re-occurs: install a `core.sshCommand` wrapper `.bat` that logs `set` to a file — git's resolution priority makes the wrapper both a working override and an env-dump in one step, so you can see exactly which `GIT_SSH` value git received.

=== Forwarding the *Linux* `ssh-agent` into Wine — why not?

Tried; doesn't work. Git-for-Windows `ssh.exe` cannot speak to a Linux `AF_UNIX` socket (Cygwin socket format mismatch). The mingw-agent-in-Wine route is the simplest path that keeps the key passphrase-protected on disk.

=== Pointing GitExtensions at the host `/usr/bin/ssh` (`Z:\usr\bin\ssh`) — why not?

A natural-looking alternative is: let the Ubuntu host's `ssh-agent` be the single source of truth, forward `SSH_AUTH_SOCK` into Wine, and tell GitExtensions / git to call the *host* `ssh` binary via Wine's `Z:` drive (e.g. `GIT_SSH=Z:\usr\bin\ssh`). It looks plausible — Wine's Z: drive exposes the Linux filesystem, the agent works (`wine 'Z:\usr\bin\ssh-add' -l` lists keys, `wine 'Z:\usr\bin\ssh' -T git@github.com` authenticates fine). But `git push` from inside GitExtensions still fails with `fatal: Could not read from remote repository`.

The reason is a fundamental Wine limitation:

* `Z:\usr\bin\ssh` is a *Linux ELF* binary, not a Windows PE binary.
* When invoked as `wine 'Z:\usr\bin\ssh' ...` from a Linux shell, the outer `wine` launcher (itself a Linux program) recognizes the ELF and just `execve()`'s it — Wine's Windows-process machinery is never involved. That's why the standalone test works.
* When `git.exe` (a Windows PE binary already running under Wine) tries to spawn the same path via `CreateProcess`, Wine's Windows process loader (`NtCreateUserProcess`) attempts to load the target as a *Windows PE* image. ELF ≠ PE, the load fails silently (`pid 0000 tid 0000` in `WINEDEBUG=+process`), and git sees a broken pipe.

Wine *does* ship an escape hatch — `start.exe /unix /path/to/linux/binary` — which can fork+exec a native Linux binary from a Windows process. Wrapping ssh in a `.cmd` shim that does `start /b /wait /unix /usr/bin/ssh %*` is enough to actually run ssh and authenticate (`ssh -vvv` shows `debug1: Exit status 0`, the agent answers, github accepts the key). But `start /unix` is designed for fire-and-forget GUI helpers and only forwards *stderr* — `stdin`/`stdout` are detached. git's pack protocol can't flow through, so the remote sees EOF and the operation aborts.

The only clean fix would be a custom mingw PE wrapper that uses Wine's internal `__wine_unix_call` to do a real `fork()` + `execve()` of the Linux ssh while inheriting all three stdio fds. That's possible but invasive enough that the in-Wine mingw `ssh-agent` route (above) remains the simplest workable design.
